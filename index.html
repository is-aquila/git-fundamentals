Serve para registrar as alterações de um repositório git em um histórico próprio.
A melhor maneira de se utilizar esse comando é realizar pequenos commits à medida que o projeto avança, focando nas funcionalidades ou correções específicas para que cada um fique mais simples de ser lido e rastreado, sendo de grande importância que se mantenha a coesão entre um commit e outro.
Também é importante a utilização das boas práticas como as regras do versionamento semântico, que permitirão uma melhor legibilidade ao histórico do código que está sendo desenvolvido, facilitando o entendimento e diminuindo a confusão que pode vir a ocorrer em códigos de maior magnitude e complexidade.

O commit salva alterações localmente, mas não altera, em nada, o repositório remoto, enquanto o push envia essas alterações para o repositório remoto. Commits compõem o histórico local do repositório na máquina do usuário, e o push é como uma continuidade desta atividade, armazenando as alterações e atualizações, contidas nos commits locais, no repositório remoto.
Os commits podem ser alterados, atualizados, corrigidos e verificados localmente sem afetar o repositório remoto, da mesma maneira que o repositório remoto pode ser alterado sem afetar o repositório local, o que significa que, mesmo que ambos contenham arquivos com diferentes características, a combinação de ambos os comandos pode fazer com que se relacionem representando um único projeto.

É um identificador que serve de referência que ajuda a apontar para um commit específico e, geralmente, é utilizado para marcar as versões do software que está sendo desenvolvido. As tags são muito úteis para identificar momentos importantes do histórico como versões específicas, alterações, correções, adição de funções, dentre outros. Também ajuda, por exemplo, a encontrar as últimas versões estáveis de um projeto, o que permite que se descubra mais facilmente em que ponto no passado um determinado arquivo ou determinada funcionalidade pode ter sido corrompida ou ter parado de funcionar.
De qualquer forma, a tag é uma ferramenta de grande importância no versionamento de softwares.

Branch é como uma ferramenta facilitadora para ramificação do desenvolvimento principal do projeto, de modo que, ao criar uma branch, é como criar uma cópia do projeto original, na qual se faz possível fazer alterações diversas sem o medo de afetar o mesmo, oferecendo a segurança de não quebrar o código original, pois os commits da branch também são armazenados em um histórico próprio, e podendo fazer um “merge” futuramente, comparando ambos projetos e definindo quais alterações conflitantes com o projeto original deverão ser adicionadas ao projeto e quais não, gerando uma versão melhorada e acrescendo o histórico original com os commits da branch.

Conflitos costumam ocorrer quando duas ou mais mudanças incompatíveis ocorrem no mesmo arquivo, não necessariamente ao mesmo tempo. A resolução do problema não é muito complexa mas exige certo cuidado, pois deve ser feita de forma manual, sendo necessário comparar os arquivos conflitantes e decidindo quais informações são relevantes ou não para o código, assim como quais podem ou não ser conflitantes com o código e, feito isso, deve-se fazer um commit para que o processo de resolução seja finalizado.
Esta é uma prática importante e que deve ser feita com cautela para evitar a quebra do código ou condições para mau funcionamento ou resultados incorretos de de outras funcionalidades ou arquivos.
